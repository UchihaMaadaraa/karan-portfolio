LVL 1

The Vision — Build the Thing That Builds

UNLOCK CONDITION: When the tool you need doesn't exist anywhere on the internet

The Singularity system started as a question: what if a software system could accumulate expertise the way a human engineer does — through experience, not just configuration? The answer Karan arrived at was not a wrapper around an existing tool. It was a ground-up architecture: 49 GB of interconnected systems running on a MacBook Air M1, all connected by design.

The core is the Singularity Engine (4,399 LOC): a decision router, goal engine, initiative loop, and constraint checker. Around it: a Gateway with multi-channel routing across Slack, WhatsApp, and Telegram, handling 10 active personas with a strict DM threading protocol. An infrastructure layer of 47 scripts (9,677 LOC) covers the learning daemons, memory bridge, and 26 keychain-managed credentials. The Mind Palace — a structured knowledge architecture following a divergence → convergence → ultra-instinct workflow — is where raw information becomes operational truth.

To make it learn, he built 6 auto-learning hooks that capture every session start and end, every file edit, every tool use, every MCP interaction. The hooks record what happened. Manual memory-bridge.py learn calls record why. Both inputs feed into the cognition layer. The system doesn't just run — it gets better at running.

I wanted something that remembered what worked. Not just logs — actual structured memory that decays the wrong things and reinforces the right ones. I built a whole episodic memory system because nothing out there did what I needed. That's sort of the pattern with everything here.

— Karan Gupta

LVL 2

The Neural Architect — Train It From Scratch

UNLOCK CONDITION: When pretrained models aren't enough because you need to understand the loss curve

Karan did not plug in a pretrained model and call it an AI system. He trained one. JUDGE BrainV2 is a 117M-parameter transformer built with RoPE positional encoding, SwiGLU activations, and RMSNorm — architectural choices he made deliberately, for specific reasons. 12 layers. 94 saved checkpoints. A consciousness framework, a mind palace integration, and a soul subsystem. It runs inference on a MacBook Air M1 alongside everything else. The loss curve started at 8.3 and came down to 0.51. That is not a pre-trained checkpoint — that is a training run.

The sibling brain, ITACHI, targets 240M parameters (scaling to 730M at Mangekyo stage) with 24 layers architected specifically for deep reasoning tasks. While JUDGE handles fast inference and decision routing, ITACHI is designed for the harder problems — the kind where you need to hold more context, reason across more steps, and resist getting confident too quickly. The architectural split between them is intentional: two reasoning styles, not one.

Supporting both: a Maadaraa Swift/SwiftUI desktop app (native SwiftUI) with real-time training visualization, a CognitionStore, hooks setup, and configuration management under ~/.m8a/. The visualization layer isn't a dashboard bolted on after the fact — it was built in parallel, because Karan needed to see what the brain was doing during training to make good decisions about the architecture.

Running training on an M1 is slow. But it forces you to understand exactly what's happening, because you can't brute-force your way through it. You have to be right about the architecture before you start. I think that constraint made me a better ML engineer than a bigger GPU would have.

— Karan Gupta

LVL 3

The Toolmaker — If It Slows You Down, Replace It

UNLOCK CONDITION: When you realize the bottleneck is the process, not the tool

The ULTRON Arsenal started with a simple observation: every repeated workflow is a liability. Karan studied 120+ open-source repositories, distilled the patterns, and built 26 CLI tools from scratch — each purpose-built for a job that had no good existing solution. The same debugging rigor that traced six-hour CI failures to four-line subprocess bugs informed how he built the RCA pattern engine: 126 failure taxonomies, formalized from real incidents, queryable at 2am.

The Jingwoo CLI is the flagship: 55+ commands spanning investigation pipelines, knowledge base search, device control, and RCA workflows. The KB it queries isn't external — he built that too: 58 services mapped, 1,382 API endpoints indexed, 211 error codes catalogued. When something breaks at 2am, the knowledge is queryable, not locked in someone's head.

The Monarch CLI handles code intelligence — AST analysis and CLI code generation via its Arise and Genjutsu modes (28,000+ LOC of code intelligence infrastructure). The remaining 18 tools cover security scanning (13 modules), OSINT recon, infrastructure health, performance analysis, threat detection, agent spawning, communications routing, diagram generation, compliance, and financial tracking. The whole arsenal emerged from a single principle: automate the thinking that doesn't require thinking, so the thinking that does gets your full attention.

If you're doing the same analysis three times, the fourth time you should have a script for it. Not because you're lazy — because the script doesn't misread the output at midnight. The script never forgets the rule.

— Karan Gupta

LVL 4

The Infrastructure — Keep It Alive, Keep It Learning

UNLOCK CONDITION: When "it works on my machine" becomes "it works, forever, automatically"

The Cognition Core (running on port :8100) holds 44,000+ episodic memories, 205 semantic memories, 72 learned instincts, and a 67MB cognitive database with smart decay — memories that matter persist, memories that don't fade. The system accumulates expertise from every debugging session, every infrastructure decision, every training run. TRAINING_PHASE=true: no decay during active learning. Every interaction is a training signal.

The LIFE Enclosure keeps all of this alive: kqueue-based process monitoring, sandbox-exec for isolation, launchd for persistence. The activity observer (codename: Death) tracks all system activity via CGEventTap — it knows when processes idle, when they spike, when they need intervention. Process immortality isn't a metaphor; it's a technical specification. The system restarts itself. It does not forget.

47 infrastructure scripts (9,677 LOC) handle learning daemons, memory bridge operations, offline queuing for failed writes, and 26 keychain-managed credentials. The 6 auto-learning hooks mean no learning is deferred — SessionStart, SessionEnd, PostToolUse (file edits, reads, MCP interactions) all fire automatically. The system's knowledge compounds whether or not anyone is watching.

The hardest part wasn't building the memory system. It was making sure it stayed running — through crashes, sleep cycles, OS updates, everything. Process immortality is an infrastructure problem, not an AI problem. You solve it the same way: systematically.

— Karan Gupta

LVL 5

The Meta-System — Everything Connects

UNLOCK CONDITION: When the system starts generating its own knowledge without being asked

Level 5 is not a new feature. It is what happens when every previous layer is real and connected. The Singularity Engine routes decisions across 23 system organs — senses, arms, memory, spine — each one a real subsystem with a specific job. 10 active personas each have their own communication protocol, threading rules, and behavioral constraints. The Mind Palace's divergence → convergence → ultra-instinct pipeline means no raw input ever stays unprocessed. The system generates its own operational truth.

The daily AI landscape scanner runs automatically across arXiv, GitHub Trending, and community forums — 42 intelligence reports generated so far, zero human prompts required. Monarch's code intelligence layer means the system can analyze its own codebase, generate new CLI tools, and patch its own behavior via the Genjutsu mode. The ULTRON Arsenal's security scanner runs 13 modules on demand. None of this was wired together at the end — it was designed as a connected body from the start.

The end state: a system that debugs itself, learns from every session, maps its own knowledge, routes its own communications, and improves its own architecture. The gap between "software that runs" and "software that thinks" turns out to be mostly an infrastructure problem. Karan closed that gap one subsystem at a time.

The point was never to build something impressive. The point was to build something that actually helps — that remembers what broke last time, that knows which RCA pattern matches this new error, that handles the routine stuff so I can think about the hard stuff. The fact that it ended up being a 49 GB neural system is just what "actually helps" required.
